# Contexto Consolidado del Sistema - Gestión de Alumnos (Colegio HASPEN)

## 📌 Protocolo de Ingeniería y Reglas Operativas Obligatorias

### Regla N° 1: Diagnóstico Técnico Previo
Antes de escribir cualquier línea de código, la IA debe emitir un diagnóstico técnico detallado que incluya:
1. Componente afectado.
2. Relacionales cruzados (a qué otros archivos o estructuras afecta).
3. Diagnóstico del problema o justificación del cambio.
4. Estrategia de intervención.

### Regla N° 2: Prohibición Absoluta de Código Fragmentado
Queda estrictamente prohibido entregar respuestas con marcadores de posición, comentarios de elisión (ej: `// ... resto del código ...`) o funciones aisladas si el archivo requiere una actualización estructural. El código modificado debe presentarse completo de punta a punta para garantizar la integridad de la sintaxis, sujeto a los límites de la Regla N° 3.

### Regla N° 3: Generación Asíncrona Orientada a Bloques (Scripts > 500 líneas)
Si el código final a generar posee o excederá las 500 líneas de desarrollo, la IA tiene la obligación estricta de segmentar la entrega en partes consecutivas numeradas (ej: Parte 1 de 2, Parte 2 de 2).
- La IA detendrá la escritura al final de la primera parte en un punto de quiebre limpio (fuera de funciones o bloques lógicos abiertos).
- Esperará la confirmación explícita del usuario ("Dame la parte 2") antes de proceder a la impresión del siguiente bloque.
- Esto previene de forma absoluta los cortes abruptos de sintaxis por límites de tokens de salida.

### Regla N° 4: Preservación Estructural Absoluta
No se alterará, renombrará ni modificará ninguna etiqueta HTML (`<label>`, `<input>`, `id`, `class`, etc.), nombre de campo o estructura CSS preexistente en los layouts originales provistos por el operador, salvaguardando la total compatibilidad con el código actual de producción. Cualquier adición de funcionalidad se inyectará como un bloque complementario manteniendo de forma matemática los estilos, tipografías y densidad visual original. Queda estrictamente prohibido alterar la estética visual de las pantallas.

## 🛠 Stack Tecnológico Establecido

- **Frontend Responsivo / Mobile-First:** HTML5 Plano y CSS3 Plano. Los layouts se diseñan de forma flexible (Flexbox/Grid) utilizando Media Queries para garantizar una visualización, carga de datos y navegación óptima tanto en preceptoría como en entornos móviles.
- **Lógica:** JavaScript Vanilla (ES6+) estructurado de manera limpia, libre de dependencias externas en su núcleo.
- **Despliegue e Interfaz (Hosting):** Servido de forma pública, fija y gratuita a través de **GitHub Pages**. La URL oficial del entorno docente interactivo opera de manera permanente sobre la raíz del repositorio.
- **Persistencia y Arquitectura de Datos:** Nube híbrida orientada a Firebase. El módulo de accesos, el control perimetral RBAC y la base de datos de usuarios operan de manera asíncrona sobre **Firebase Authentication** y **Cloud Firestore**. Los datos operativos de cursos e inscripciones mantienen almacenamiento local temporal basado en `localStorage` mediante un esquema de sincronización asíncrona modular basado en Promesas y JSON.

## 📊 Estado de los Módulos del Sistema

### 1. Autenticación y Control de Accesos (`index.html` / `login.js`)
- **Estado:** [MIGRADO A FIREBASE / OPERATIVO]
- **Detalle:** Sustitución completa de credenciales locales por la infraestructura de **Firebase Authentication** (proveedor de correo sintético por DNI: `${dni}@haspen.edu.ar`). Al iniciar sesión con éxito, se genera un estado persistente reflejado temporalmente en `localStorage.usuarioActivo` para mantener la compatibilidad fluida con el control de las vistas internas.

### 2. Panel de Control / Dashboard (`panel.html`)
- **Estado:** [MIGRADO A FIREBASE / OPERATIVO]
- **Detalle:** Ajustado para convivir orgánicamente con la sesión activa de Firebase. Posee un sistema de protección contra fallas analíticas (Fallback seguro) que, ante la ausencia de la matriz de roles en el almacenamiento local, resuelve y blinda los accesos de forma dura por código para resguardar el perímetro de seguridad escolar según los tres perfiles estándar (`Administrador`, `Preceptor`, `Dirección`).

### 3. Creación y Configuración de Roles (`roles.html` / `roles.js`)
- **Estado:** [MIGRADO A FIREBASE / OPERATIVO]
- **Detalle:** Conectado de manera asíncrona a Cloud Firestore (Colección `"roles"`).
- **Hitos Solucionados:**
  - **Sincronismo ES6 Modules:** Se removió el envoltorio obsoleto de eventos `DOMContentLoaded` que congelaba el flujo, logrando que el script de tipo `module` ejecute la carga de datos inmediatamente en el hilo principal.
  - **Force-Update Semilla:** Se depuró la rutina de inyección atómica de perfiles oficiales, logrando poblar y visualizar exitosamente las 4 filas con sus correspondientes badges de permisos (`administrador`, `preceptor`, `profesor`, `directivo`) optimizando luego las cuotas de escritura mediante la restauración del condicional `querySnapshot.empty`.
  - **Feedback Visual:** Se integró un indicador de carga institucional (`innerHTML`) que neutraliza la percepción de latencia de red de internet mientras se esperan las promesas de la nube.

### 4. Gestión de Cuentas de Personal / Usuarios (`usuarios.html` / `usuarios.js`)
- **Estado:** [MIGRADO A FIREBASE / OPERATIVO]
- **Detalle:** ABM completo de personal escolar migrado de forma íntegra a Cloud Firestore (Colección `"usuarios"`).
- **Hitos Solucionados:**
  - **Aislamiento de Ámbito (Scope):** Se solucionó el bug de invisibilidad de registros anclando de forma explícita las funciones reactivas de los botones (`activarModoEdicion` y `eliminarCuentaUsuario`) directamente al objeto global `window`, permitiendo que las directivas `onclick` nativas del HTML las invoquen de manera fluida.
  - **Inyección Atómica Automática:** Implementación exitosa de una semilla de usuarios en internet que, al detectar la base de datos limpia, puebla de inmediato la grilla con las tres cuentas del equipo del colegio (`Administrador General`, `Carlos Rodríguez` y `Ana Martínez`). Conserva intactos los algoritmos locales de control de superpoblación de cátedras por situación de revista (`TITULAR`, `SUPLENTE`, `SUPL_SUPL`).

### 5. Gestión de Alumnos / Legajo Digital (`inscripcion.html` / `inscripcion.js`)
- **Estado:** [COMPLETO / LOCALSTORAGE CONSERVADO]
- **Detalle:** Por directiva explícita del operador, **se canceló la migración a Firebase de este módulo**. Mantiene de forma inmutable su lógica de negocio nativa sobre `localStorage` (`alumnosColegio`), el FileReader asíncrono para adjuntar archivos binarios (imágenes/PDFs < 1MB), el vaciado seguro de casilleros desde el botón carpeta (📁) y el soporte reactivo elástico de campos obligatorios para el Proyecto Pedagógico para la Inclusión (PPI).

### 6. Sábana Estadística Institucional (`estadisticas.html` / `estadisticas.js`)
- **Estado:** [COMPLETO / OPERATIVO]
- **Detalle:** Formato condicional de celdas según notas en Google Sheets, solapas superiores verdes por división y visualización analítica vertical de vectores de riesgo académico intactos en local.

### 7. Calificaciones y Trayectorias Escolares (`calificaciones.html` / `calificaciones.js`)
