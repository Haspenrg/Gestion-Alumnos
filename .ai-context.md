# Contexto del Proyecto: Gestión Alumnos (Colegio HASPEN)

## 📌 Visión General
Este sistema centraliza la administración escolar del Colegio HASPEN. Permite gestionar el alta de personal, inscripciones de alumnos, calificaciones y estadísticas institucionales.

## 🛠️ Arquitectura y Tecnologías
- **Frontend**: HTML5, CSS3 nativo (estilos.css) y JavaScript asíncrono (ES6+ módulos).
- **Backend de Autenticación y Permisos**: Firebase Authentication y Cloud Firestore.
- **Persistencia Operativa**: Híbrida. Los datos críticos de usuarios residen en la nube, mientras que las inscripciones y estadísticas se manejan de forma local/inmutable en `localStorage`.

## 🔐 Reglas de Autenticación e Identidad (Actualizado)
1. **Identificador del Operador (UI)**: El usuario siempre interactúa visualmente utilizando su número de **DNI** como credencial de acceso y recuperación en la interfaz de usuario.
2. **Resolución Híbrida de Credenciales**: El sistema no utiliza correos sintéticos (`@haspen.edu.ar`). El flujo de código toma el DNI ingresado, consulta el documento correspondiente en Cloud Firestore y extrae la dirección de **correo electrónico real/personal** del agente.
3. **Validación en Firebase Auth**: La autenticación contra el servidor de Firebase se ejecuta utilizando estrictamente dicho **correo real** obtenido de la base de datos de internet junto con la contraseña ingresada.
4. **Recuperación Autónoma Activa**: El flujo de restablecimiento de contraseña (`sendPasswordResetEmail`) despacha de forma nativa el enlace de recuperación a la bandeja de entrada del **correo real** del operador tras validar la existencia de su DNI en Firestore. El Administrador debe garantizar la sincronización exacta del campo `email` entre Firestore y Firebase Authentication.

## 📋 Reglas Operativas Obligatorias para la IA (Mantenidas)
1. **Diagnóstico Técnico Previo**: Antes de entregar cualquier propuesta de código, se debe proveer obligatoriamente un desglose con: Componente afectado, Relacionales cruzados, Diagnóstico del problema y Estrategia de intervención.
2. **Prohibición de Código Fragmentado**: Entregar siempre los scripts y archivos modificados de manera completa, de punta a punta. Está estrictamente prohibido usar comentarios de elisión como `// ... resto del código ...`.
3. **Generación Asíncrona por Bloques**: Si la extensión de un script o layout supera las 500 líneas de código, se segmentará obligatoriamente en partes y se detendrá la respuesta, esperando la confirmación explícita del usuario (ej: "Dame la parte 2") antes de continuar.
4. **Preservación Estructural Absoluta**: Está terminantemente prohibido alterar IDs, nombres de clases, etiquetas HTML estructurantes o la estética visual/layouts definida en los archivos provistos por el usuario.
